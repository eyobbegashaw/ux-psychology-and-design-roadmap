 
This file operationalizes the psychology of attention capture. Attention is a zero-sum game. Our design isn't competing with other apps; it's competing with the user's own internal thoughts, their environment, and their deep-seated desire to do nothing. We must understand the two attentional systems to ethically and effectively gain entry into the user's consciousness.

**1. Bottom-Up vs. Top-Down Attention (The Two Doorways to the Brain)**
- **Bottom-Up Attention (The Involuntary Interrupt):** This is a System 1, automatic, stimulus-driven capture. It's a primal reflex. A sudden movement, a bright color against a dull background, a flashing notification, a loud sound. It's uncontrollable.
    - *Dev Implementation:* This is the domain of motion and contrast. A new element entering the DOM with a CSS animation (`@keyframes slideIn`) triggers bottom-up attention. A high-contrast CTA button (e.g., bright orange on a muted grey background) is a bottom-up magnet. The `Notification` API on mobile/browser is the ultimate bottom-up tool—an interrupt that appears outside the product itself. This power must be used with extreme restraint. Every unnecessary interrupt is an act of psychological violence against the user's focus.
- **Top-Down Attention (The Voluntary Searchlight):** This is a System 2, goal-directed, effortful search. The user is actively looking for something—a specific menu item, a tracking number in a confirmation email, a key metric on a dashboard.
    - *Dev Implementation:* This is the domain of information architecture, search, and user-controlled customization. A well-implemented `Command palette` (like `Cmd+K` in a developer tool or advanced SaaS) is a pure top-down attention interface. The user knows what they want and is actively filtering out all other stimuli. Poorly-implemented top-down design is a navigation menu that forces the user to think and search. Our design must be built to serve a user in *both* modes, often simultaneously.

**2. The Salience Pyramid: Guiding the Gaze**
We can't make everything pop out, or nothing will. We must build a clear, intentional hierarchy of visual salience, a pyramid where only one element sits at the very top.
- **Level 1 (The Apex - The Primary Action):** The single most important visual element on the page. High contrast, surrounded by a generous moat of white space, motion applied strategically (e.g., a subtle pulse on load, then it stops). This element must be the unambiguous "next cognitive beat."
- **Level 2 (The Support - Data and Navigation):** Still visible but toned down. Lower contrast colors, smaller typography, in predictable, scannable locations (e.g., a left-hand nav bar, a top filter bar). This information is available to top-down attention when needed but is not screaming for bottom-up capture.
- **Level 3 (The Muted - Legal and Utility):** The lowest salience. Low-contrast grey text, smallest size. Footer links, copyright info, terms of service. This is critical information that must be findable but must never interrupt the primary flow.
- *Dev Specification:* The front-end component library must have semantic CSS utility classes that map to these levels: `.visual-weight-primary`, `.visual-weight-secondary`, `.visual-weight-tertiary`. A developer choosing a visual weight class is making a psychological declaration about the attentional hierarchy. The system design kit (e.g., Figma tokens) must enforce these three levels and only these three, preventing a designer from accidentally creating a visual arms race where a secondary support element is competing with the primary CTA.

**3. The "Surprise and Delight" as an Attentional Reset**
Human attention is a detector of change. When a UI becomes predictable, it becomes invisible (a phenomenon known as banner blindness). A perfectly-timed micro-interaction that breaks the pattern in a positive way can reset attention and create a memorable emotional spike.
- *Example:* After a user completes a deeply boring, complex setup wizard, a simple, charming confetti animation can jolt them out of cognitive fatigue and create a moment of joy. This isn't fluff; it's a deliberate attentional and emotional intervention timed at a known pain point (the end of a tedious task), leveraging the Peak-End Rule from folder 04.
- *Dev Note:* This must be architected as a rare, precious event in the system. A `SurpriseAndDelightService` should have a strict cooldown period. The confetti animation that was delightful the first time is annoying the tenth time. It must be a non-repeating, contextually-aware event, not a default success state. It's an attentional defibrillator, not a pacemaker.
